# Collection Services 演讲稿（优化版）

## 开场
大家好，我是今天的讲者。接下来我会用几分钟带大家认识全新的 Collection Services 演示版本，重点聚焦在“纯前端存储”和“多窗口实时协同”两大亮点。

## 核心亮点一：信息全部落地在 localStorage
1. 我们把原本散落在后端的账户资料、申请表单、审批记录，统统迁移到了浏览器的 localStorage。
2. 这样的做法带来三大好处：
   - **零依赖后端**：即使断网或没有服务器环境，系统依旧能运作；
   - **快速体验**：所有读写都发生在本地，响应几乎是即时的；
   - **数据可视化**：开发者可以直接在开发者工具中检视、导出、调试数据。
3. 技术上我们封装了统一的 `storage` 工具层：
   - `getStoredUsers`、`persistUsers` 等方法负责读取/写入用户；
   - `setCurrentUser`、`logout` 保证登录状态可控；
   - 所有方法都会先判断是否在浏览器环境，再安全地序列化/反序列化数据，避免 JSON 解析报错。

> **强调点**：这是一个完全自给自足的认证/审批流，无需数据库、无需接口，适合做快速 PoC 或线下路演。

## 核心亮点二：多窗口互不干扰又能及时同步
1. 因为所有状态都写进 localStorage，只要浏览器还在，数据就共享给所有标签页。
2. 操作演示：
   - 左边标签页以“用户”身份登录，提交一份申请；
   - 右边标签页以“管理员”身份登录，点击刷新即可读取同一份申请；
   - 管理员审批完成后，用户页面刷新立刻看到审批结果。
3. localStorage 在刷新时自动读取最新数据，配合我们解耦的路由系统，实现“多角色、多窗口”互不干扰：
   - 用户页、管理员页、审核页各自的路由状态互相隔离；
   - 但数据层共用同一个本地存储，因此切换或刷新都会看到最新同步结果；
   - 如果需要更强的实时性，可以在下一阶段加入 `storage` 事件监听，即时推送 UI 更新。

## 架构简述（30 秒带过）
- **前端组件层**：表单负责收集输入、触发服务方法；
- **服务封装层**：`authService` 等模块读写 localStorage，并输出统一的数据模型；
- **可拓展空间**：保留 Supabase 客户端接口，未来若要切换至云端，只要替换存储实现即可。
